# 程序大型化协作指引

日期：2026-06-14

## 0. 适用口径

当前优先版 N0-N5 已完成非美术交付前压力测试。本指引从 N5 之后接续，用于把项目从“当前优先纵切”推进到“完整版大型生活模拟”的程序侧实装阶段。

本指引不替代 `doc/当前优先设计/21_程序极速协作指引.md`。前者服务完整版扩围，后者服务已完成的当前优先版冲刺。

执行红线：

1. 美术资源、最终点位美化和表现层替换不作为程序侧阻塞项；只要求资源 key、数据 ID 和运行入口稳定。
2. 新增内容不得只停留在 Markdown，必须落到结构化数据、运行入口、守卫测试或浏览器 smoke 之一。
3. 地区迁移继续只允许通过街景 gate、subregionGate、码头、城门、驿站、商路节点或明确交互点触发；总览地图只保留调试和路线提示能力。
4. 工艺系统必须持续连接材料、工坊、NPC、缺陷、返修、订单、作品履历和长线经营，不能退化为孤立按钮小游戏。
5. 每个大节点必须按本文写明的固定团队数启动协作。节点完成后必须同步文档、跑验证、提交 Git，并保持工作区干净。

## 3. M0.9 全地区生活规格入轨

固定团队数：5。

团队分配：

| 团队 | 审计范围 | 主要输出 |
| --- | --- | --- |
| A 集成与数据基线 | 当前状态树、数据表、测试基线、冻结 ID | 完整版扩围的代码入口清单和风险清单 |
| B 地区与 NPC 生活循环 | 11 区小地区、NPC、活动、订单、节令、结局回声 | 每区“生活链 / 人物链 / 活动链 / 订单链”缺口表 |
| C 工艺与作品反馈 | 旗舰工艺、工序、缺陷、返修、导师、订单拒收 | 工艺深度差距和 M2 可复用输入模型建议 |
| D 地图与正式移动 | 人工地图 JSON、街景点位、gate、subregionGate、总览红线 | 地图运行时缺口和移动守卫补强建议 |
| E 经济、引导与 QA | 长局经济、库存、维护费、教程、异常态、浏览器 smoke | 长局模拟、引导与错误态优先级矩阵 |

必须交付：

1. 新增或更新 M0.9 入轨记录，列出 11 个大地区的程序缺口矩阵。
2. 明确 M1 第一批要入库的地区章节、NPC、活动、订单、工艺和测试入口。
3. 明确哪些缺口属于表现层，哪些属于程序侧必须推进。
4. 若发现已有代码与完整版红线冲突，先补守卫；若只是内容密度不足，进入 M1 排期。
5. 完成后至少跑 `npm test`；若改动运行时代码，还需跑 `npm run build` 与内置浏览器 smoke。

完成闸门：

- 5 个团队均给出回查结论。
- 文档中每个地区都有下一步程序动作，而不是笼统写“后续补充”。
- 工作区完成提交并保持干净。

## 4. M1 全地区核心章节入库

固定团队数：6。

团队分配：

1. 集成与结构团队：统一章节数据结构、入口状态、存档迁移和守卫。
2. 江南 / 巴蜀团队：把既有主轴提升为可长期回访章节样板。
3. 岭南 / 黔滇团队：补生活链、商贸链和民族工艺链。
4. 荆楚 / 赣鄱 / 徽州团队：补水路、瓷纸墨与文房荐藏章节。
5. 京畿 / 三晋 / 雪域 / 西域团队：补宫造、票号、唐卡、玉作和远行章节。
6. QA 与浏览器团队：守卫入口、章节推进、跨区路线、结局读取和内置浏览器 smoke。

每个地区最低验收：

- 1 条核心章节线。
- 至少 3 类玩法入口：工艺 / 生活或修习 / 商贸路线或地方活动。
- 至少 1 位本地匠人、1 位路线或商贸人物、1 位生活或文化人物参与。
- 至少 1 个订单或后续委托能读取作品状态。
- 至少 1 条 reducer、数据完整性或浏览器 smoke 守卫。

## 5. M2 工艺深度与半实时桥接

固定团队数：5。

团队分配：

1. 工艺状态机团队：抽象火候、湿度、刀线、张力、染液、锤击、阴干等输入结果。
2. 旗舰工艺数据团队：优先覆盖龙泉剑、青瓷、蜀锦、香云纱、苗银、楚漆、景德镇瓷、宣纸、景泰蓝、推光漆、唐卡、玉作。
3. UI 与输入桥团队：把按钮校准升级为可替换半实时输入层。
4. NPC 与返修反馈团队：师傅现场复核、鉴评、订单拒收、返修履历读取同一套诊断结果。
5. QA 团队：覆盖成功、失败、返修、导师介入、订单拒收和旧档兼容。

完成闸门：

- 至少 3 门旗舰工艺形成专属半实时输入桥样板。
- 其余旗舰工艺有同构数据字段，不阻塞后续逐门补小游戏。
- 工艺结果能进入作品实例、NPC 反馈、订单、结局和售卖。

## 6. M3 跨区供应链与长线经济

固定团队数：5。

团队分配：

1. 解锁树与地区推进团队。
2. 跨区供应链与路线风险团队。
3. 工坊、家园、仓储和维护费团队。
4. 价格、库存、声望和商誉平衡团队。
5. 长局模拟与回归 QA 团队。

完成闸门：

- 至少跑通多季长局压力测试。
- 断供、维护费、库存膨胀、通货膨胀、订单收益和路线稳定有可读报告。
- 玩家跨区扩张有资源理由，而不是只靠主线按钮解锁。

## 7. M4 开放世界生活层

固定团队数：7。

团队分配：

1. NPC 日程与关系事件团队。
2. 节令与地方活动团队。
3. 家园来访、徒弟和社区团队。
4. 地图运行时与点位工具团队。
5. 内容数据库与可视化工具团队。
6. 教程、可达性和异常态团队。
7. QA、浏览器和长线存档团队。

完成闸门：

- 11 区至少都有可回访 NPC 日程、地区活动和生活事件。
- 玩家能在无主轴强推时自然找到工艺、人物、路线和生活目标。
- 内容工具能发现缺 NPC、缺活动、缺点位、缺订单、缺结局回声等问题。

## 8. M5 云存档与异步社交预研

固定团队数：3。

团队分配：

1. 后端边界与账户方案团队。
2. 存储适配、云存档和冲突处理团队。
3. 安全、隐私、排行榜和异步社交 QA 团队。

执行原则：

- M5 不得提前打断 M1-M4 的单机生活模拟闭环。
- 先做接口边界和存储适配，不急于接真实服务。
- 所有联网功能都必须能在本地单机降级。

## 9. 每个节点的共同验收格式

每个节点完成时必须记录：

1. 启动了多少团队，是否符合本文固定团队数。
2. 改了哪些程序文件、数据文件、测试文件和文档文件。
3. 新增了哪些可运行入口、自动化守卫和浏览器 smoke。
4. 哪些设计仍未实装，为什么不阻塞本节点。
5. 本节点提交 hash、验证命令和遗留风险。

禁止交付：

- 只写文档、不接数据或入口。
- 用总览地图迁移替代街景通道。
- 把美术缺图当作程序不推进的理由。
- 新增没有冻结 ID 或没有被引用的内容表项。
- 完成节点后不提交或留下脏工作区。
