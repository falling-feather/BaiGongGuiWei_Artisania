## 1. 当前判断

项目已经不是早期 demo，核心规则、地区框架、资源生产、订单交付、NPC 交互、节令活动、夜市、地区声望、珍品阁来访、专属联作、返修失败和结局报告等链路已经可以运行。

但项目也**不能按“除了美术资源和位置调整之外都完成”交接**。如果按照类端游体量、类似《东方夜雀食堂》《星露谷物语》的生活模拟密度来验收，当前仍有不少非美术缺口。下一支代码团队应把这些缺口视为主任务，而不是表现层润色。

## 2. 已经基本跑通的开发链路

### 2.1 主循环与存档

- 新建/继续本地存档、多槽位、玩家名号、设置与基础 HUD 已接入。
- 核心规则集中在 `src/engine/reducer.ts`，通过 Zustand 与 React/Phaser 桥接。
- 存档版本、旧档迁移、地图状态、田圃、订单、NPC 关系、家访与代表作等状态已经进入统一状态树。

### 2.2 地区、移动与街景

- 全国 11 个大地区和小地区骨架已存在。
- 当前仍保留大地图/区域地图，但它主要用于总览和调试。
- 正式游戏迁移原则已经明确：玩家应在街景中走到通道、出口、商路节点或交互点后按 `E` 触发跨区移动，不能把“点击地图直接传送”作为正式体验。
- 街景交互、NPC、工坊、采集点、商贩、子区通道和提示清理已有运行链路；`App` 现会用当前街景 `buildRegionSpec` 校验 `interact-subregion-gate` / `interact-gate`，避免总览或伪造事件绕过街景通道规则。人工地图运行时接入已从 `runtimeLayoutFromEditorSnapshot` 适配器和 `RUNTIME_MAP_LAYOUTS` 样板起步，并把 11 个大地区代表小地区迁成 `src/data/mapLayouts/*.json` 检入资产，相关道路、点位、NPC 站位和玩家起点已进入 `StreetScene`；镇务小地区卡片现在会展示本地内容摘要但仍不提供点击迁移。完整编辑器 JSON 文件、全部小地区点位、图层和季节变体还没有完全收口。

### 2.3 资源、工艺与订单

- `resources`、`industries`、`crafts`、`regionContent`、`craftInteractions` 等数据已形成“原料 -> 半成品 -> 成品 -> 订单/售卖/赠礼/陈列”的主链。
- 多门代表工艺已经有 `CraftInteractionSpec`、工序选择、缺陷、返修、师承 NPC、工坊升级和代表作口碑加成。
- 目前工艺更像“数据化工序 + 选择结算”，还不是完整的实时动作式工艺模拟。

### 2.4 NPC、活动与人物线

- NPC 功能门槛、切磋、护商危机、断供危机、夜市摆摊、节令活动、主动订单、寄售/赊账和家访已接入。
- 节令摊位链已从秦淮灯市、西域绿洲巴扎、赣鄱瓷镇开窑会、巴蜀茶马会扩到岭南骑楼夜市；茶马会与骑楼夜市均已有三轮活动挑战、三阶段摊位、客群/组合/策略、封账选择和后续订单。
- 珍品阁/藏客等回访链已经能把代表作、地区声望、后续订单和结局段落串起来。
- 外地 NPC 密度、长期日程、人物线后续分支和地区生活事件仍不够“端游化”。

### 2.5 文本与结局

- 文本已经不再只是占位：代表作命名、地区声望、人物关系、订单描述、失败返修、协作结局和家访段落都有实装。
- 仍需要继续把文档中的地区人物、工艺设定和事件链补进结构化数据，并通过测试守卫防止“只有文案、没有运行入口”。

## 3. 非美术未完成项

这些是代码团队接手后要明确记录和推进的缺口，不能被归为“等美术”。

1. **真实工艺模拟深度不足**
   大多数工艺仍以按钮、选项和规则结算为主。当前已开始把 `CraftInteractionStage` 推成可测试的阶段诊断状态机：作品会记录每段控制/风险/稳平险结果，缺陷与返修履历也能保留病根工序，订单拒收和 NPC 鉴评已能读取这些病根，师傅授艺后也可生成规则层现场复核记录。后续仍应继续抽象火候、湿度、刀线、张力、染液、锤击、阴干、返修窗口等更细状态，并逐步给旗舰工艺加入实时或半实时操作。

2. **全地区内容密度不均**
   11 个地区都有设计规格和代表内容，但 NPC 长线、节令活动、后续订单、回访、冲突事件和地区结局密度仍不足。目标是每个地区都能支撑独立生活循环，而不是只作为材料产地。

3. **人工地图与运行时接入刚进入样板阶段**
   地图编辑器能导出 `artisania-map-editor/v2` JSON，运行时也已能通过 `runtimeLayoutFromEditorSnapshot` 读取首批检入 JSON 布局资产，覆盖 11 个大地区代表小地区的道路、工坊/资源点、活动点、NPC 站位、子区通道、跨区通道、玩家起点和简易碰撞装饰。仍需把完整编辑器 JSON 文件加载、瓦片层、门/碰撞层、季节/天气变体、调试标记和全部小地区精确点位接入同一稳定管线。

4. **正式移动规则仍需守护**
   大地图只能保留为总览/调试。正式迁移必须走街景出口、商路节点和交互点。需要补数据守卫或调试开关，避免未来代码重新引入“点地图传送”的正式入口。

5. **经济与长局平衡不足**
   当前能跑通订单和售卖，但还缺长周期压力测试：季节资源、商路风险、库存膨胀、金币通胀、师徒成长、珍品阁口碑、地区声望和失败惩罚之间的数值平衡。

6. **数据完整性工具还不够**
   已有部分数据守卫，但仍要补全：每地区必须有小地区、NPC、活动、工坊、工艺、资源链、移动入口、代表订单、失败返修、家访/结局段落；每个新增 ID 必须被至少一个运行入口引用。

7. **教学、可达性与异常态不足**
   需要补新手链路、失败提示、无材料/无订单/背包满/存档迁移异常等状态。现在主流程可运行，但“玩家不知道下一步做什么”的风险仍高。

8. **后端与页游化能力未启动**
   账号、云存档、异步社交、排行榜、分享报告、多人或轻社交都还没有工程实现。它们不是当前最急，但不能在最终规划里遗漏。

## 4. 接手优先级

### P0：稳住基线

- 先执行 `npm run typecheck`、`npm test`、`npm run build`。
- 阅读 `git status --short` 和关键 diff，确认哪些改动来自上一轮开发。
- 不做大规模重构，不回退用户或其他智能体留下的改动。

### P1：补非美术核心系统

- 把 `CraftInteractionSpec` 推进成真正可驱动 UI、失败、返修、熟练度和导师反馈的工艺状态机；当前 `CraftStageOutcome` 已覆盖阶段控制/风险/缺陷归因，且已被订单拒收、NPC 鉴评和 `CraftMentorIntervention` 读取，下一步要把这些结果继续接入更多工艺分支、师傅站位演出和半实时小游戏。
- 为 11 个地区补齐内容数据密度和运行入口，优先补 NPC 日程、地区活动、后续订单和回访线。
- 完成地图编辑器 JSON 到运行时街景的稳定接入。
- 为正式移动规则建立调试/正式模式边界。

### P2：补测试与内容守卫

- 为每个新链路补 reducer 测试或数据完整性测试。
- 增加全地区覆盖测试，保证文档设计不会停留在 Markdown。
- 增加长局模拟或压力测试脚本，观察经济、资源和订单是否失衡。

### P3：再处理表现层

- 美术资源套用、最终 tiles/buildings/props 替换、人物动画精修和区域点位美化放在最低优先级。
- 只在某个系统必须靠资源 key 才能验收时，做最小映射或占位资源接入。

## 6. 交接验收标准

代码组每完成一个大切片，至少要满足：

- 玩家能从街景入口触发，而不是只能在调试面板里调用。
- 状态变化能进入存档，旧档不会崩。
- 文档中的新增设定有对应结构化数据和运行入口。
- 至少有一个自动化测试覆盖主要成功路径；高风险分支要覆盖失败路径。
- 若暂时无法完成，应在 [当前进度与规划差距盘点.md](当前进度与规划差距盘点.md) 写明缺口、风险和下一步。

---

## 7. 不要先做的事

- 不要先投入大量时间替换最终美术资源。
- 不要把地图点击传送包装成正式玩法。
- 不要只加文案而不接入运行条件、状态变化或测试。
- 不要为了“看起来完成”跳过数据守卫。
- 不要做跨模块重构来追求整洁，除非它直接解除当前系统缺口。
