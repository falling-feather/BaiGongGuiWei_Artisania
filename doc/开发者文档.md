# 百工归位 · Artisania — 开发者文档

> 本文目标：让任何一位新接手的程序员 / AI，**无需口头交接**即可独立续开发本项目。
> 读完本文你应当能回答：代码放在哪、出问题先看哪、各模块怎么连、加内容/加功能从哪下手。

## 0. 一分钟速览

- **是什么**：以中国非物质文化遗产「手艺工序」为核心的 **2D 像素风俯视街道游览** 游戏。玩家操控角色在街道（未来拓展到城市 / 国家地图）上自由行走，走到不同的手艺体验点亲手体验非遗工艺。手艺背后是一套隐藏的「传承 / 市场 / 生活 / 精神」四维数值与经济系统，支撑探索与经营。
- **技术栈**：React 18 + TypeScript 5 + Vite 5；**游戏世界用 Phaser 3**（像素风俯视、tilemap、物理碰撞）；状态管理用 Zustand；测试用 Vitest；界面样式为纯 CSS。
- **架构一句话**：**纯函数引擎（engine，隐藏数值/经济）+ 注入式内容（data）+ Zustand 桥接（store）+ Phaser 游戏世界（game）+ React HUD/弹窗（components）+ 可替换存档（storage）**。
- **核心原则**：游戏规则（数值结算）**只**存在于 `src/engine/reducer.ts`。Phaser 只管「空间与移动」，React 只管「HUD 与弹窗」，二者都不直接改规则。

## 1. 启动与常用命令

```powershell
npm install        # 安装依赖
```

## 3. 数据流：一次玩家操作如何贯穿全栈

```
玩家在街上走近「蓝染坊」，按 E
      │
      ▼
StreetScene.ts  →  emitBus({ type:'interact-craft', craftId })   // Phaser 只发事件，不碰规则
      │
      ▼
App.tsx  →  onBus 监听 → setActiveCraftId(craftId)            // React 接管，弹出体验面板
      │
      ▼
CraftExperienceModal 点「亲手制作」
      │
      ▼
store/gameStore.ts  →  gameReducer(state, action, content)     // 纯函数求新状态
      │                         └─ 读取 content（crafts/apprentices/events）
      ▼
set({ state: next })  +  localStorageAdapter.save(next)        // 更新 + 异步存档
      │
      ▼
HUD / 弹窗订阅的切片自动重渲染（四维、手艺状态…）
```

> **提示事件**：玩家走进/离开体验点感应范围时，StreetScene 会 `emitBus({type:'hint', text})`，
> App 监听后把提示文本传给 `Hud`（「按 E 进入…」）。Phaser 与 React 间的所有通信都走 `game/EventBus.ts`。

**关键不变量**：
1. 规则只在 `reducer.ts`。改数值平衡、加结算逻辑都来这里。
2. `gameReducer` 是纯函数，**无副作用**（不碰 localStorage、不 fetch）。副作用（存档）由 store 在 reducer 返回后执行。
3. 内容是**注入**的（`GameContent` 作为第三参数传入），引擎本身不 import 任何具体手艺/事件，因此引擎可独立测试、可移植到后端。
4. **Phaser 不碰数值**：游戏世界只负责空间/移动/触发，手艺的数值结算一律走引擎。

## 4. 核心类型速查（详见 `engine/types.ts`）

| 类型 | 作用 |
|---|---|
| `MetricKey` | 四维键：`heritage` 传承 / `market` 市场 / `life` 生活 / `spirit` 精神 |
| `Metrics` | `Record<MetricKey, number>`，取值 0–100 |
| `MetricZone` | 区间评价：`crisis/low/healthy/high/excess`，健康区 30–70 |
| `ProcessStep` | 一道工序：资源/人力成本、正常影响、是否可省略、省略影响 |
| `Craft` | 一门手艺：工序链 `processChain`、四维基线、联动标签、风险、`outputResourceId`（产出的成品资源键，供应链终端） |
| `Apprentice` | 学徒：性格特质、所学手艺、熟练度、留存意愿 |
| `GameEvent` / `EventChoice` / `GameEffect` | 随机事件、选项、结构化效果 |
| `CraftState` | 某手艺在本局的动态状态（四维、是否解锁、产出数） |
| `GameState` | **单一事实来源**：种子、回合、四维、资源、手艺、学徒、待处理事件、日志、状态、报告、`unlockedRegions`、`currentRegion`、`achievements`、`seenStory`（已看过的剧情节点）、`flags`（叙事标记，驱动分支）、`playerName`（玩家名号）、`devMode`（开发者模式）、`npcAffinity`（`Record<npcId,number>` 好感度）、`completedQuests`（已交付任务 id） |
| `GameReport` | 结局命运报告：标题/概述/最终四维/存续手艺/亮点/`epilogue`（个性化尾声，呼应名号与拉择走向） |
| `ResourceDef` / `ResourceTier` | 资源定义与分层（raw 原料 → material 半成品 → product 成品）；product 可选 `value`（订单交付基准售价） |
| `IndustryDef` / `MiniGameType` | 基础产业定义与其采料小游戏类型（六种） |
| `RegionDef` / `SubregionDef` / `TerrainTheme` | 大地区档案、区内小地区与地貌主题；`currentRegion` 指大地区，`currentSubregion` 指区内探索地点 |
| `AchievementDef` | 成就定义：`id/name/desc` + `predicate(state)=>boolean`（纯函数） |
| `StoryBeat` / `StoryChoice` | 剧情节点：`id/title/lines/trigger(state)=>boolean` + 可选 `choices`（每个 `StoryChoice` 含 `setFlags?`/`logMessage?`） |
| `NpcDef` | NPC：`id/name/role`(`vendor`关联人物/`tourist`游客)/`regionId` + 可选 `anchorCraftId`(驻守的手艺点)/`greetings[]`；江南样板已扩展 `functions`、`preferences`、`relationshipLines`、`intel`、`personalDilemma`、`endingInfluence` |
| `QuestDef` | 任务：`id/npcId/title/desc/requireAffinity`(好感门槛) + `condition(state)=>boolean`纯函数 + `reward`(coin/metrics) + `completeLog` |
| `TerrainKind` | 地貌主型：`water`(河流+石桥)/`mountain`(山石)/`coast`(海岸)/`plain`(街巷)，由 `terrainKind()` 从地区地貌推导 |
| `RegionNpcSpec` | 场景层 NPC 生成规格（id/name/role/anchorCraftId），由 `buildRegionSpec` 下发 |
| `GameAction` | 所有可派发动作（见下表） |

### GameAction 一览

| action | 含义 | 由谁派发 |
|---|---|---|
| `NEW_GAME{seed?,playerName?}` | 开新局（名号决定剧情称呼，`fallingfeather` 触发开发者模式） | MainMenu「开启新局」/ 结局「再启新局」 |
| `RUN_PROCESS{craftId,skipStepIds}` | 按工序链开工，可省略部分工序 | CraftPanel |
| `TAKE_ORDER{craftId}` | **交付订单**：消耗 1 件成品库存换收入，售价随该手艺传承品质浮动（`orderPrice`）；无成品则拒绝 | CraftExperienceModal |
| `HOLD_EXHIBITION` | 举办市集/展演 | App「镇务」 |
| `RESOLVE_EVENT{choiceId}` | 处理当前随机事件 | EventModal |
| `END_TURN` | 结束本季，结算/补给/抽事件 | App「结束本季」 |
| `GATHER_RESOURCE{industryId,quality?}` | 在当地跑一项基础产业，quality 来自小游戏手感 | IndustryPage（整页内嵌小游戏） |
| `TRAVEL{regionId}` | 前往一个已解锁大地区，并进入该大地区默认小地区 | 场景出入口（WorldMapModal 仅总览，RegionPanel 仅提示路线） |
| `TRAVEL_SUBREGION{subregionId}` | 在当前大地区内切换小地区 | RegionPanel「区内小地区」 |
| `UNLOCK_REGION{regionId}` | 按 `RouteSpec` 路资与门槛解锁相邻地区 | 场景出入口（WorldMapModal 仅总览，RegionPanel 仅提示路线） |
| `SEEN_STORY{storyId,choiceId?}` | 标记某剧情节点已看；带 `choiceId` 时把该抉择的 `setFlags` 写入 `flags` 并记日志 | StoryModal |
| `TALK_NPC{npcId}` | 与 NPC 攀谈，好感度 +8（封顶 100），按好感解锁 `NpcIntelDef` 并写入 `knownTopics`/`flags` | NpcDialogModal |
| `USE_NPC_FUNCTION{npcId,functionKind,itemId?}` | 使用 NPC 功能入口；按 `npcFunctions.ts` 的好感门槛和每日限制结算授艺、路线、订单、联作、鉴评、家园来访，联作/鉴评需要作品实例 | NpcDialogModal |
| `COMPLETE_QUEST{questId}` | 交付任务：校验好感门槛 + `condition(state)`，达标则发放 `reward`(coin/metrics) 并记入 `completedQuests` | NpcDialogModal |

> **成就检测**：顶层 `gameReducer` 是 `checkAchievements(reduce(...))` 的包装——每次 action 结算后
> 遵一遍 `content.achievements` 的 `predicate(state)`，新达成的追加进 `state.achievements` 并写日志。仍为纯函数。

## 4.5 2D 游戏世界（Phaser 层）

> 这一层是本次重做的核心：把原文本玩法包进一个可自由行走的像素世界。
> 设计上**刻意全部用程序生成的占位图**，方便日后逐个替换为真实美术（见「美术替换点」）。

### 模块职责

| 文件 | 职责 |
|---|---|
| `game/PhaserGame.tsx` | React 容器组件。`useEffect` 里 `new Phaser.Game(config)` 挂到 div，卸载时 `game.destroy(true)`。开发态把实例挂到 `window.__game` 供浏览器调试。 |
| `game/gameConfig.ts` | Phaser 配置：`pixelArt:true`、800×500 初始分辨率、`RESIZE` 铺满舞台（避免高窗口黑边）、arcade 物理（无重力）、`scene:[StreetScene]`。 |
| `game/EventBus.ts` | 单例 `Phaser.Events.EventEmitter`，**双向**：`emitBus`/`onBus`（Phaser→React：`interact-craft`/`interact-industry`/`interact-gate`/`interact-npc`(NPC 对话)/`hint`/`scene-ready`/`player-pos`(玩家瓦片坐标+地图尺寸)/`region-points`(小地图点位)/`zoom-changed`(当前倍率)）与 `emitCommand`/`onCommand`（React→Phaser：`enter-region` 附 `RegionMapSpec`、`zoom` 附 `delta`/`absolute`）。`RegionMapSpec` 现含 `terrain:TerrainKind`（`water`/`mountain`/`coast`/`plain`）与 `npcs:RegionNpcSpec[]`。 |
| `game/regionSpec.ts` | `buildRegionSpec(regionId, state)`：把数据层+当前状态翻译成场景可用的 `RegionMapSpec`（大地区名、小地区名、地面配色、产业点含层级、已实现手艺点、相邻地区出入口）。场景因此与数据层解耦。 |
| `game/textures.ts` | **★美术替换点★** 先由 `preloadArtTextures(scene)` 加载 `public/assets/game/**` 下的 PNG：地面/道路变体、河流边缘/内部、建筑变体、角色行走序列帧、特效；再由 `generatePlaceholderTextures(scene)` 兜底生成缺失纹理（幂等，已加载则跳过）。`TILE=32`。新增键包括 `groundMoss`/`groundStone`/`groundSoil`、`roadVertical`/`roadCross`、`waterEdgeLeft`/`waterEdgeRight`、多类 `craft*`/`ind*` 建筑、`npcTourist`/`npcVendor`。 |

### 完整 demo 数据流（地图切换 / 小游戏 / 资源传递）

```
场景就绪 → emitBus('scene-ready')
  → App 收到后 buildRegionSpec(currentRegion) → emitCommand('enter-region', spec) → 场景铺图

走近产业点按 E → emitBus('interact-industry', industryId)
  → App 开 IndustryPage 整页（右栏内嵌 MINIGAME_REGISTRY 机制）→ 玩家「出手」得 quality(0..1)
  → dispatch GATHER_RESOURCE{industryId, quality} → 引擎结算：扣原料/工时，产出 = round(yield*(1+quality))

走近出入口按 E → emitBus('interact-gate', {regionId, unlocked})
  → 已解锁 → dispatch TRAVEL；未解锁 → dispatch UNLOCK_REGION(扣30文)，成功则接着 TRAVEL
  → currentRegion / unlockedRegions 变化 → App 的 store 订阅检测到 signature 变化 → 重新 emitCommand('enter-region') → 场景切图

资源传递：resources 是全局单一池，跨区携带即「运输」。
  例：江南缺煤 → 解锁邻区赣鄱(有 coal) → 采煤 → 回江南冶铁(铁矿本地+煤) → 锻剑成品。

相机缩放 / 小地图（V0.5.12）：
  +/- 键·小键盘加减·Shift+滚轮·小地图按钮 → 场景 `applyZoom` 限幅 1~4×。
  玩家移动→`player-pos`→App 状态→`Minimap` 实时渲染玩家点与坐标；切区重建→`region-points`→小地图重画点位。
```

> 产业分三层：采集(harvest，仅耗工时，由地区 `localResources` 授权) → 精炼(refine，原料→材料) → 制作(product，材料→成品)。
> 采集业的可用性在引擎 `gatherResource` 里特判：`input` 为空且 `output ∈ region.localResources` 即放行，无需写进 `region.industries`。

### 关键约定 / 踩坑记录

- **不要用 `React.StrictMode`**：它会双挂载，导致 Phaser 生成两个 canvas。`main.tsx` 里已移除（注释说明了原因）。
- **单场景胜于 Boot+切换**：早期 `BootScene → StreetScene` 的 `scene.start` 静默不触发，已合并为单一 `StreetScene` 在 `create()` 里自己生成贴图。新增场景时把它加进 `gameConfig` 的 `scene` 数组。
- **`window.__game` 调试钩子**：仅开发态（`import.meta.env.DEV`）挂载，可在浏览器控制台直接读写玩家位置/速度，测试移动与交互。
- 交互点感应范围 `INTERACT_RANGE = TILE*1.8`；进入范围发 `hint`，按 E 按点类型发 `interact-craft`/`interact-industry`/`interact-gate`。若 NPC 与点位同在范围内，`nearbyPriority` 会让 NPC 优先触发，避免驻守人物被店铺入口遮蔽。

### 如何新增一个交互点（产业 / 手艺 / 出入口）

地图内容**完全由数据驱动**，新增点不改场景：

1. **产业**：在 `data/industries.ts` 加一条 `IndustryDef`；把 id 写进某地区 `region.industries`（采集业则只需让其 `output` 出现在该地区 `localResources`）。
2. **手艺**：在 `data/crafts.ts` 加一门手艺，并把其 id 放进某地区 `signatureCrafts`（只有已实现的 craft 才会出现在地图上）。各地区招牌工艺已用 `sigCraft(...)` 工厂批量补齐（`SIGNATURE_CRAFTS`，41 门，统一「备料选材(可省略)→核心工序A→核心工序B」三段链），蓝染/竹编另有详尽手写链作样板。
3. **出入口**：在 `data/regions.ts` 给地区互填 `neighbors`，并在 `data/regionContent.ts` 补对应 `RouteSpec`。`buildRegionSpec()` 会把路线名、路资、门槛提示下发给场景出入口；正式玩法中靠近出入口按 E 前往或开通。
4. 配色：地区 `terrain.palette` 前两色作为 [地面色, 街道色]，场景运行时 `setTint`。

### 地图编辑器试装（image_test）

由于自动识别切图无法稳定理解建筑语义与摆放意图，本分支先加入一个人工地图编辑器，用来让策划/美术直接输出可复核的地图数据。入口：启动 Vite 后打开 `http://127.0.0.1:5173/?editor=1`。`App.tsx` 会在该模式下直接渲染 `components/MapEditor.tsx`，不启动 Phaser 游戏场景和存档副作用。

编辑器当前是 DOM 工作台，核心能力：

1. 左侧素材栏按地面、道路、水体、建筑、道具、单位、标记分类，像植物大战僵尸选卡片一样选择当前画笔。
2. 中央网格点击绘制；瓦片类元素直接写入格子，建筑/道具/单位以对象层放置，并显示鼠标跟随预览。
3. 右侧属性面板可调整地图尺寸、对象占格宽高、坐标、碰撞箱、同格同层是否允许堆叠、交互类型；建筑可切常态/夏景/雨天/秋景/冬雪外观，单位可预览方向、静态帧与行走帧播放。
4. 堆叠规则由 `allowSameCellStack` 控制：地面瓦片默认不可同格叠同类；建筑默认会替换同格同层旧建筑；街景道具/NPC/标记默认允许叠加，便于在草地、道路或水岸上组合装饰。
5. 点击任意格子后，右侧「地块堆叠」会按 z 值从上到下列出覆盖该格的大模型，可直接选中、删除或把当前模型插入该格，解决大型建筑擦除只命中左上角的问题。
6. 「模型装配器」可从空白画布导入建筑主体、冬季雪顶、季节装饰、街景组件或自定义图片路径；每个组件都是独立图层，可调 X/Y、缩放、透明度、图片路径和图层顺序，再给整体模型设置分类、渲染层、占格、碰撞箱、堆叠规则与交互类型。
7. 导出格式为 `artisania-map-editor/v2`，包含 `tileSize`、`size`、`modelDefinitions[]`、`tiles[]` 与 `objects[]`。导出的对象会保留 `itemId`、`asset`、`modelLayers`、`layer`、`tileW/tileH`、`solid`、`interaction`、外观状态、动画参数和堆叠标记，后续可接入 Phaser Tilemap 或 `RegionMapSpec` 的人工布局字段。

后续要扩展素材，只需要把 PNG 放进 `public/assets/game/...`，再在 `MapEditor.tsx` 的 `PALETTE` 里补一条 `tile(...)`、`object(...)` 或 `character(...)`。运行时地图仍以 `StreetScene` 为准，编辑器导出的 JSON 暂作为验收与人工设计输入，等第一地区布局稳定后再把 JSON 接入 `regionSpec.ts` / `StreetScene.buildRegion()`。

## 4.7 玩家名号 / 开发者模式 / 剧情分支系统

> 这一层给 demo 添上叙事与代入感：玩家取名→剧情称呼→抉择分叉。全部以**数据驱动 + 引擎纯函数 selector** 实现，引擎逻辑零硬编码。

### 玩家名号与开发者模式（V0.5.3）

- 开局页输入名号→`NEW_GAME{playerName}`→`createInitialState(...,playerName)` 存入 `state.playerName`。
- `state.ts` 常量 `DEV_NAME = 'fallingfeather'`；`devMode = playerName.trim().toLowerCase() === DEV_NAME`。
- **开发者模式**：`unlockedRegions` = 全部地区、`resources` = `DEV_RESOURCES`（各 999999）；END_TURN 时 `labor` 不衰减（保持海量工时）。纯函数判定，不开后门。

### 剧情节点与抉择分支（V0.5.4 / V0.5.5）

| 模块 | 职责 |
|---|---|
| `data/story.ts` | **剧情表** `STORY_BEATS`。每个 `StoryBeat` 用 `trigger(state)` 纯函数决定何时出现（按回合/成就/已看节点/`flags`/解锁地区数/开发者模式）；有 `choices` 的为抉择节点。`renderStoryLine(line,name)` 把 `{name}` 替换为玩家名（空名回退「无名匠人」）。 |
| `engine/reducer.ts` 的 `nextStoryBeat(state,content)` | 选取首个 `!seenStory.includes(id) && trigger(state)` 的节点；无则返回 null。纯函数 selector，不改状态。 |
| `engine/reducer.ts` 的 `SEEN_STORY` | 追加 storyId 进 `seenStory`；带 `choiceId` 时从 `content.story` 找到节点与抉择，把 `choice.setFlags` 并入 `flags`、推 `choice.logMessage`（支持 `{name}`）。 |
| `components/StoryModal.tsx` | 读 `nextStoryBeat`；`pendingEvent` 存在或无节点时返回 null。有 `choices` 则逐项渲染按钮派 `SEEN_STORY{storyId,choiceId}`；否则单一「继续」派 `SEEN_STORY{storyId}`。 |

**分支机制（零引擎逻辑改动）**：`GameEffect.setFlags` + `StoryChoice.setFlags` 写入叙事标记 → 后续 `StoryBeat.trigger` 读 `flags` 分叉。示例：「立心」节点选「守正」→`oath-tradition`，选「趋时」→`oath-market`；第 4 季分到 `path-tradition` / `path-market`。随机事件「审美风向突变」选项写 `chased-trend`/`kept-tradition`，趋流触发「风过之后」回响节点。

**App 挂载**：`{!tutorialOpen && <StoryModal />}`——引导期不出剧情；剧情与随机事件互斥（`pendingEvent` 优先）。

### 个性化结局尾声（V0.5.7）

- `generateReport` 调 `generateEpilogue(state, m)` 写入 `GameReport.epilogue`。
- 根据 `flags` 的 `oath-tradition`/`oath-market` 走向，叠加最终传承/市场度，给出「体面回答 / 退了几步」或「热闹是真热闹 / 没等到回暖」的呼应或反差；再叠 `kept-tradition`/`chased-trend` 的风向回响句，玩家名号嵌入正文。
- `GameOverReport` 渲染 `.report__epilogue`（朱砂左边线、分段缩进）。纯函数，不读外部数据。

### 手艺工序链 / 供应链终端（V0.5.8）

> 打通「采料(raw) → 精炼(material) → 手艺(product)」三层：手艺 RUN_PROCESS 现在**消耗半成品、产出成品**。

| 环节 | 说明 |
|---|---|
| `Craft.outputResourceId` | 手艺的成品资源键（product 层）。蓝染→`indigoCloth`、竹编→`bambooWare`。缺省则不产实物。 |
| 工序 `resourceCost` | 改用 material 键（`indigoVat`/`bambooSplit`），与精炼产业的 `output` 对接。 |
| `reducer.ts` 的 `runProcess` | 汇总工序耗料→校验人力/物料→扣除→产出 1 件 `outputResourceId` 入 `resources` 池，日志取成品名（由 `content.resources` 注入）。 |

#### 订单经济闭环（V0.6.0）

| `CraftExperienceModal` | 产出行显示「库存 N」与「订单价 X 文」；接订单按钮无成品时禁用并提示。 |
| `GameContent.resources` | 新增可选资源词典，仅用于日志/展示取名；由 `gameStore` 注入 `RESOURCES`。 |
| `state.ts` 初始/开发者资源 + 每回合补给 | 同步改为 material 键（`indigoVat`/`bambooSplit`）。 |
| `CraftExperienceModal` | 新增耗料/产出预览与不足预警；料/工不足时禁用「亲手制作」。 |

## 4.8 Demo 终章四模块（V0.7.0）

### 模块 1 · 地貌差异化（terrain）

- `data/quests.ts`（`QuestDef`）：关联人物发布，`requireAffinity` 好感门槛 + `condition(state)` 纯函数（如「库存 bambooWare≥1」），达标 `COMPLETE_QUEST` 领 `reward`。`questsForNpc(npcId)` 取任务。
- `engine/reducer.ts`：
  - `TALK_NPC{npcId}`：`npcAffinity[npcId] += AFFINITY_PER_TALK(8)`，封顶 `AFFINITY_MAX(100)`；按关系阶段选台词，达到 `NpcIntelDef.unlockAffinity` 后把见闻写入 `npcStates[npcId].knownTopics/revealedIntelIds` 与 `flags`。
  - `GIFT_ITEM{itemId,npcId}`：作品赠礼会读取 `NpcGiftPreference`，按资源、工艺、产地、描述词和品质给好感加成或轻微降权。
  - `USE_NPC_FUNCTION{npcId,functionKind,itemId?}`：读取 `NpcDef.functions`，按 `npcFunctions.ts` 的门槛与每日限制结算：授艺给属性/见闻，路线写 `route:*` 情报，订单提升市场，联作提升作品品质并写 `collaboratorNpcIds`，鉴评写题跋，家园来访写长期 flag。
  - `COMPLETE_QUEST{questId}`：校验未完成 + 好感达门槛 + `condition`，发奖并记入 `completedQuests`；不足则仅记日志。
- `NpcDialogModal.tsx`：名/役 + 关系阶段 + 人物功能标签 + 人物矛盾 + 地方见闻 + 功能行动 + 偏好提示 + 好感度条（`.npc-affinity`）+ 任务交付按钮 + 攀谈/赠礼/告辞。`App.tsx` 路由 `interact-npc` → 此弹窗。

### 模块 4 · 点击寻路移动（pathfinding）

- `StreetScene` `pointerdown`（忽略右键）→ `requestPathTo(worldX,worldY)`：把世界坐标转瓦片，`nearestWalkable` 找最近可走格（BFS 半径≤3），`findPath(start,goal)` 4 向 BFS 返回瓦片队列（不含起点），显示 `moveTarget` 光环。
- `update()` 沿 `path` 逐格行进（`PLAYER_SPEED=160`），抵达即弹出队首；键盘移动会打断点击寻路。
- 可走性来自 `blocked[ty][tx]`：地形（水/树/石）与建筑占格在 `buildRegion`/`buildTerrain` 中 `markBlocked`，桥面放行。

## 6. 如何扩展内容（最常见的开发任务）

### 加一门新手艺
1. 在 `src/data/crafts.ts` 的 `CRAFTS` 数组里新增一个 `Craft` 对象：填 `id`、`name`、`region`、`blurb`、`processChain`（每道工序的成本与四维影响）、`baseMetrics`、`synergyTags`、`risks`。
2. 若希望开局就有它的动态状态，确认 `createInitialState` 会为每个 craft 生成 `CraftState`（默认已遍历传入的 crafts）。
3. 完成。引擎与 UI 自动识别，无需改其它文件。

### 加一个随机事件
1. 在 `src/data/events.ts` 的 `EVENTS` 数组新增 `GameEvent`：`id/title/description/choices[]/weight`。
2. 每个 `choice.effect` 用 `GameEffect` 描述对四维/资源/某手艺的影响和日志文本。
3. `weight` 越大越容易被抽中。

### 调整平衡数值
- 初始资源、每回合补给、人力上限：`engine/state.ts`（`INITIAL_RESOURCES`、`LABOR_PER_TURN`）与 `engine/reducer.ts`（`TURN_RESOURCE_REGEN`、收益公式 `incomeBase`、`REGION_UNLOCK_COST`）。

### 加一条成就
1. 在 `src/data/achievements.ts` 的 `ACHIEVEMENTS` 加一条 `AchievementDef`：`id/name/desc` + `predicate(state)`（纯函数，只读 `GameState`）。
2. 完成。reducer 每次结算后自动检测解锁，成就册与 toast 自动识别。

### 加一个地区
1. 在 `src/data/regions.ts` 加一条大地区基础档案，并在 `REGION_SUBREGIONS` 给它配置 2–4 个 `SubregionDef`（主街、市集、郊野、工坊、码头等）。
2. 在 `src/data/mapLayout.ts` 给大地区补一条 `{x,y}` 坐标；大地图只展示这个大地区节点，不展示小地区。
3. 互填大地区 `neighbors` 打通商路；区内切换走 `TRAVEL_SUBREGION`，不参与商路解锁成本。

### 加一个剧情节点 / 分支
1. 在 `src/data/story.ts` 的 `STORY_BEATS` 加一条 `StoryBeat`：`id/title/lines` + `trigger(state)`（纯函数，只读 `GameState`，可读 `flags`/`seenStory`/`turn` 等）。
2. 要分支则加 `choices`，每个 `StoryChoice` 用 `setFlags` 写叙事标记；后续节点的 `trigger` 读该标记分叉。
3. 若要随机事件也能打标记，在 `data/events.ts` 的 `choice` 上加 `setFlags`。引擎/UI 不改。

### 给一门手艺做整页主题（可选精修）
> 注意：**新增手艺无需任何额外步骤即已是整页**——`getCraftPageTheme` 会自动生成通用主题。以下仅用于把某门手艺从「自动主题」升级为「定制主题」。
1. 在 `src/components/craftPageThemes.ts` 的 `CRAFT_PAGE_THEMES` 加一条以 craft id 为键的主题：主视觉占位文案、tagline、故事段落、tips。该条目会覆盖自动主题。
2. 无需改 `App.tsx` 或路由——`getCraftPageTheme` 优先取定制主题，缺省回落自动主题。
3. 整页玩法（工序省略/制作/接单）复用引擎 `RUN_PROCESS`/`TAKE_ORDER`，无需改引擎。主视觉/立绘/背景按「美术资源设计.md」逐页绘制后替换占位即可。

### 加一个 NPC / 任务
1. NPC：在 `src/data/npcs.ts` 的 `NPCS` 加一条 `NpcDef`（`vendor` 需填 `anchorCraftId` 驻守某手艺点；`tourist` 落街游走）。除 `greetings` 外，优先补 `functions`、`preferences`、`relationshipLines`、`intel`、`personalDilemma`，让 NPC 能提供地方知识和差异化赠礼反馈。
2. 任务：在 `src/data/quests.ts` 的 `QUESTS` 加一条 `QuestDef`：`npcId` 关联发布人、`requireAffinity` 好感门槛、`condition(state)` 纯函数交付条件、`reward`（coin/metrics）、`completeLog`（支持 `{name}`）。
3. 完成。`NpcDialogModal` 与引擎 `TALK_NPC`/`GIFT_ITEM`/`USE_NPC_FUNCTION`/`COMPLETE_QUEST` 自动识别，场景不改；只有新增互动规则时才改 reducer 或 `npcFunctions.ts`。

### 加一种地貌
1. 在 `src/game/EventBus.ts` 的 `TerrainKind` 加一个新型；`regionSpec.ts` 的 `terrainKind()` 加关键词映射。
2. 在 `StreetScene.buildTerrain(kind)` 加该型的铺设分支（用 `addDecorTile` + `markBlocked` 摆地貌与阻挡）。
3. 玩法/数据不变；注意新地貌不要压住交互点列（见 §4.8 已知设计问题）。

## 7. 存档与未来后端

- 当前实现：`LocalStorageAdapter` 把整份 `GameState` 序列化进 `localStorage` 多槽位，带 `version` 字段（当前 `SAVE_VERSION=6`；因 `GameState` 新增 `currentSubregion` 而从 5 提升）。
- 多槽键位：`artisania:saves:index` 保存槽位顺序，`artisania:saves:active` 保存当前活跃槽，单槽数据写入 `artisania:saves:slot:{slotId}`。主菜单当前限制最多 5 个本地槽。
- 兼容旧档：旧版单槽 `artisania:save` 会在第一次 `listSaves()/load()` 时迁入 `slot-1`；旧状态若缺少 `currentSubregion`，会自动进入该大地区默认小地区。
- `StorageAdapter` 现在提供 `save/load/listSaves/deleteSave/getActiveSlotId/setActiveSlotId/hasSave`。`gameStore.dispatch()` 每次状态变更后自动保存到活跃槽；`MainMenu` 的新建/继续/覆盖/删除都经这一层。
- **接后端时**：实现一个 `RemoteStorageAdapter implements StorageAdapter`（多槽 list/save/load/delete 改为 HTTP 调用），在 `gameStore.ts` 替换 `localStorageAdapter` 即可。账号/排行榜/云存档都从这一层接入，UI 与引擎零改动。

## 11. 给下一位接手者的提醒

- 改规则只动 `reducer.ts`，别把逻辑写进组件或 Phaser 场景。
- 保持引擎纯净：不要在引擎里 import 具体数据或调用浏览器 API。
- **Phaser 只管空间/移动，所有手艺数值结算走引擎**；Phaser↔React 只通过 `game/EventBus.ts`。
- **绝不加 `React.StrictMode`**（会让 Phaser 生成双 canvas）。
- 美术是增量替换的：从 `game/textures.ts` 的占位图逐个换成真实像素（见第 4.5 节「美术替换点」）。
- 中文文件用 UTF-8（无 BOM）；PowerShell 下避免用 `Set-Content`/`Get-Content` 处理含中文的文件，优先用编辑器工具。
- 提交信息格式：`V{大}.{中}.{小} 中文说明`。
