# 极速推进工作流

## 1. 推进原则

当前阶段只做能让项目更快交付的事：

1. 先跑通链路，再加密内容。
2. 先稳定 ID，再对接美术。
3. 先补主轴五区，再补骨架六区。
4. 先写数据守卫，再扩大表项。
5. 除非直接解除阻塞，不做跨模块大重构。

每次改动都必须落在“文档 -> 数据 -> 规则/UI -> 测试 -> 进度记录”这条链上。

## 2. 单任务标准流程

每个内容任务按以下顺序执行：

1. 在 `doc/当前优先设计` 或对应交接文档中确认范围。
2. 写入或调整 `src/data` 的结构化数据。
3. 接入 reducer、scene、modal、HUD 或资料卡入口。
4. 增加最小测试：数据守卫、reducer 链路、scene spec 或浏览器烟测记录。
5. 更新交接文档，说明已完成、暂缓和下一步。

禁止只写文案不接入口，也禁止只接入口不留验收记录。

## 3. 冲刺顺序

### P0 稳住基线

- 确认新分支构建、测试、浏览器烟测正常。
- 清理或记录本地临时文件，不把 dev server 日志和缓存纳入版本。
- 把总览地图迁移能力继续限定为调试，不作为正式游玩路径。

### P1 主轴闭环

- 江南：新档入门、街景通道、基础工艺、秦淮灯市、NPC 见闻和售卖链。
- 巴蜀：茶马路线、蜀锦/竹编、路线订单和节令摊位反馈。
- 岭南：骑楼夜市、香云纱、海贸船期、后续订单。
- 赣鄱：景德镇瓷、开窑会、复烧/返修、瓷行回访。
- 西域：绿洲巴扎、玉作、远行风险、玉柜藏客。

每个地区完成时，都要有一条 reducer 全链路测试或等价数据守卫。

### P2 骨架区补线

- 黔滇、荆楚、徽州、京畿、三晋、雪域只补最低闭环。
- 每区至少确认：小地区、代表工艺、NPC、活动或订单入口、资料卡、结局段落。
- 已经存在的长链只做断点修补，不追加大型分支。

### P3 交付前压力测试

- 新档到结局的最短通关路径。
- 继续档跨地区移动路径。
- 订单、售卖、返修、珍品阁、夜市和结局报告的组合路径。
- 长局库存、金币、声望、工时和路线稳定的基础压力检查。

## 4. 并行协作方式

如需开子智能体或分工，按以下角色拆分：

| 角色 | 负责范围 | 输出 |
| --- | --- | --- |
| 内容数据 | 地区、NPC、活动、订单、文本入库 | `src/data` 改动和文档记录 |
| 规则链路 | reducer、工艺、返修、经济、结局 | 单元测试和链路测试 |
| 场景交互 | 街景通道、HUD、modal、资料卡 | 浏览器烟测和控制台检查 |
| QA 守卫 | ID 引用、覆盖矩阵、长局压力 | 测试报告和缺口清单 |

主线分支上每次合并前必须有人从另一个角度复核：内容改动看运行入口，规则改动看文本反馈，场景改动看测试守卫。

## 5. 每日收口格式

每天收口只记录四件事：

1. 今天完成了哪些可运行链路。
2. 哪些内容从完整版被裁剪或暂缓。
3. 哪些 ID、资产 key 或测试守卫已经稳定。
4. 明天最短路径要打通什么。

记录位置优先使用：

- `doc/当前优先设计/03_优先内容验收表.md`
- `doc/代码组交接与收口清单.md`
- `doc/当前进度与规划差距盘点.md`
