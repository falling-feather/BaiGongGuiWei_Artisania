# 程序极速协作指引

更新时间：2026-06-16

这份文档是程序组最后一次集中收尾的唯一分工入口。给团队发本文即可开工；旧的 N0-N5、M1.x 文档只作为证据链和问题追溯，不再作为任务分配入口。

## 0. 使用方式

团队先读本文，不要先翻 90 多份历史文档。

阅读顺序：

1. 先看“进度比例”和“硬红线”。
2. 再看自己负责地区的“11 区收尾表”。
3. 按“一次性收尾团队”和“最终工作计划”领取任务。
4. 遇到 ID、测试、地图、旧节点来源问题时，再按“问题查找表”打开指定文档。

本轮只收程序非美术：最终美术替换、瓦片细化、云端/社交/排行榜、正式内容编辑器不纳入硬闸。

## 2. 程序现状总览

已稳定：

- 前端运行：React + Phaser + Vite。
- 状态结构：Zustand + `engine/reducer.ts` 纯函数规则结算。
- 存档：localStorage 多槽、版本迁移、旧档补齐。
- 地区：11 个大地区、43 个小地区。
- 地图：43/43 小地区已有运行时人工地图 JSON。
- 移动：正式迁移走 `StreetScene -> EventBus -> App` 的街景 gate / subregionGate / route gate。
- 章节：`REGION_CHAPTERS` 已覆盖 11 区，每区有核心章节、玩法支柱、人物角色、订单/回访/联作/护商钩子和 gap。
- 审计：`buildRegionChapterAudit` 已能检查未知引用、路线落点、人工地图、`smokeBindings`、作品状态读数和 NPC 关系读数。
- 工艺：工序、材料、缺陷、返修、师傅授艺、师傅现场复核、专注校准、手法选择、工坊升级、空间扩建和季维护均已接入。
- 订单：活动订单、NPC 订单、采办、信用、押金、逾期、作品缺陷拒收和后续续单已跑通多地区样板。
- NPC：攀谈、赠礼、授艺、订单、鉴评、家访、藏客回访、结局后日谈已成体系。
- 活动：秦淮灯市、绿洲巴扎、赣鄱开窑会、巴蜀茶马会、岭南骑楼夜市等多阶段活动已验证。
- 结局：读取 11 区、NPC 关系、活动、作品售出、藏客、护商、断供等回声。

仍需最后收口：

- 6 个 `needs-expansion` 地区要补人物回访、订单差异、材料反馈或经济统计。
- 非江南 10 区要推广多入口 `smokeBindings`。
- Ready 地区要补体验短板，不能只靠“状态 ready”过线。
- 长局经济要集中压测。
- 浏览器 QA 要复验 11 区入口和大地图负向。

## 3. 硬红线

- 不得用总览地图、资料卡、镇务面板直接迁移；正式移动必须走街景 gate 或明确交互点。
- 新内容必须同时具备数据入口、运行入口和测试/审计守卫。
- 不得只写 Markdown 宣称完成。
- 不得重命名冻结 ID。
- 不得把最终美术、瓦片细化、角色立绘缺口写成程序阻塞。
- 不得扩大到云端、排行榜、异步社交或正式内容编辑器。
- 实时动作小游戏不作为本轮硬闸，但工艺规则层字段、缺陷、返修、订单读取和反馈必须稳定。

## 6. 最终工作计划

### F0 绿基线锁定

团队：T0、T11。

目标：

- 确认当前未提交程序改动全部有测试。
- 跑 `npm run typecheck`、`npm test`、`npm run build`。
- 更新本文和 M1 总账，不再新建分散计划。

通过标准：

- 三条命令全绿。
- 工作区剩余改动可以解释到文件级。
- 没有测试失败、没有新增孤立 Markdown。

### F1 六个 needs-expansion 章节收尾

团队：T1-T6。

目标：

- 每个地区至少关闭一条人物回访、订单差异、材料反馈或经济统计链。
- 每个地区新增或扩展一组专项测试。
- 每个地区明确是否可从 `needs-expansion` 转为 `chapter-ready`；不能转的，必须只剩 M3/美术/数值类后置缺口。

回头审查：

- T0 统一比较 6 区完成密度，防止某区只补地图入口、某区补成完整章节而风格失衡。

### F2 ready 地区补短板

团队：T7。

目标：

- 巴蜀补临邛铁料订单回流和茶马会后日谈。
- 岭南补佛山铁器、端砚/石湾陶订单差异和骑楼船期后日谈。
- 赣鄱补瓷土/釉料读数、河运长线和多入口绑定。
- 西域补巴扎商路风险、织坊藏客回访和订单差异。
- 江南只做 M1.29 关系读数回归，不再扩范围。

通过标准：

- 不新增大系统。
- 每区至少一条现有入口被更深读取，而不是只加新表项。

### F3 多入口 smokeBindings 与导航审计

团队：T8、T10。

目标：

- 把江南 M1.28 的 `smokeBindings` 模式推广到其余 10 区。
- 每章至少绑定核心小地区、代表活动、代表工艺、关键 NPC、正式 route landing case。
- 保证大地图、资料卡、镇务面板仍不能绕过街景正式迁移。

通过标准：

- `regionChapterAudit` 可统计并检查每章入口覆盖。
- 导航相关测试覆盖所有新增 binding。

### F4 长局经济与压力测试

团队：T9。

目标：

- 新增 30-60 回合模拟，覆盖多区订单、维护费、库存、押金、利钱、声望和路线稳定。
- 明确过强/过弱的经济点，只做保守数值修正。

通过标准：

- 无负库存、无无法恢复的金钱死局、无路线稳定无限崩坏。
- 关键经济 flags 能被结局或后续订单读取。

### F5 最终浏览器复验

团队：T10，T0 复核。

目标：

- 新档进入江南并完成一条完整链。
- 从江南经正式 gate 到至少 3 个外区。
- 抽查 11 区 `chapterSmoke` + `smokeSubregion`。
- 验证大地图点击不迁移，只给路线提示。
- 记录控制台 error/warn。

通过标准：

- 浏览器无阻塞错误。
- 任何无法自动化的 UI 问题都要记录为明确缺口，不得口头放过。

### F6 收口锁定

团队：T0、T11。

目标：

- 全量测试、构建、文档状态一致。
- 标记哪些地区已转 `chapter-ready`。
- 输出最终程序剩余项：只允许是美术、数值、云端/社交或内容工具链。

通过标准：

- 程序组可宣布“一次性收尾完成”。
- 后续团队不再以程序结构缺口为理由拖延美术与最终 QA。

## 7. 交付格式

每个团队只交五件事：

1. 关闭了哪个 gap。
2. 改了哪些文件。
3. 新增或扩展了哪些测试。
4. 跑了哪些命令或浏览器 smoke。
5. 还剩什么，属于 P0、P1 还是后置。

禁止交付：

- 只写文档，不接数据、入口或测试。
- 只加数据，不加审计守卫。
- 用总览地图迁移代替街景 gate 可达性。
- 重命名冻结 ID。
- 把最终美术缺口写成程序阻塞。
