# 内容数据库与文本入库设计

## 1. 文档目的

当前优先版需要把文本设定推进到可入库、可检索、可运行入口引用的状态。这里的“数据库”不是马上接后端数据库，而是先把所有策划内容整理成稳定的数据表口径，继续落在现有 `src/data` 文件中，后续再迁移到编辑器或外部表格。

本阶段仍属于策划审核，不开始代码改造。审核通过后，程序只按本文件补结构化数据和守卫。

## 2. 现有数据入口

当前项目已有较完整的数据分层，当前优先版不另起炉灶：

| 内容 | 现有入口 | 策划职责 |
| --- | --- | --- |
| 大地区与小地区 | `src/data/regions.ts` | 决定地区定位、子地区名称、路线感和代表工艺 |
| 小地区可玩内容 | `src/data/subregionContent.ts` | 决定每个小地区开放哪些工坊、产业和运行入口 |
| 地区活动、夜市、路线活动 | `src/data/regionContent.ts` | 决定活动阶段、摊位、客群、策略、结算和后续订单 |
| 工艺真实工序 | `src/data/craftInteractions.ts` | 决定工序、缺陷、返修、导师和作品诊断文本 |
| NPC 与见闻 | `src/data/npcs.ts` | 决定人物身份、站位、日程、功能、好感台词和地方见闻 |
| 订单与任务 | `src/data/quests.ts` | 决定订单目标、奖励、最低品相、路线或地区反馈 |
| 百工志与资料卡 | `src/data/loreEntries.ts` | 决定文本数据库条目、检索关键词、行脚目标和引导提示 |
| 人工布局样板 | `src/data/mapLayouts/*.json` | 决定资源点、工坊、NPC、活动点、通道和跨区入口的对象 key |

## 3. 当前优先版入库范围

### 3.1 主轴五区

江南、巴蜀、岭南、赣鄱、西域需要达到“内容入库完成”：

1. 每区至少 1 条完整活动链。
2. 每区至少 1 门代表工艺有真实工序、缺陷、返修和导师反馈。
3. 每区至少 2 个关键 NPC 有功能、日程或见闻。
4. 每区至少 1 条订单或后续订单能读取地区活动/工艺结果。
5. 每区至少 3 条百工志文本：地区、工艺、路线或生活事件。
6. 每区至少 1 个可美术对接的街景布局样板。

### 3.2 骨架六区

黔滇、荆楚、徽州、京畿、三晋、雪域需要达到“世界不断链”：

1. 地区与小地区存在。
2. 代表工艺存在，且至少有一条工艺交互规格或可引用样板。
3. 至少 1 个代表 NPC 存在，并能生成见闻或功能反馈。
4. 至少 1 条百工志地区资料卡存在。
5. 结局报告能读取该地区声望或代表事件。
6. 美术组能从文档知道该区第一眼识别点和首批资产包。

## 4. 文本入库模板

### 4.1 地区条目

| 字段 | 写法 |
| --- | --- |
| `id` | `region-{regionId}-{topic}` |
| `category` | `region` |
| `regionId` | 对应 `regions.ts` 的大地区 ID |
| `subregionId` | 如指向具体小地区则填写 |
| 标题 | 地区名 + 具体场所或生活主题 |
| 摘要 | 一句话说明“玩家为何要去这里” |
| 正文 | 只写可帮助游玩的信息，不写百科式长段落 |
| 解锁 | 初见地区、NPC 见闻、完成活动或制作代表作 |

地区条目必须能回答：这里有什么、怎么去、能做什么、和哪门手艺或路线相关。

### 4.2 工艺条目

| 字段 | 写法 |
| --- | --- |
| `id` | `craft-{craftId}-{topic}` |
| `category` | `craft` |
| `craftId` | 对应 `crafts.ts` / `craftInteractions.ts` 的工艺 ID |
| `regionId` | 代表产地 |
| 重点 | 材料、工序、风险、缺陷、返修、导师 |

工艺文本不写成“介绍某非遗项目”，而要写成玩家能理解的操作知识。例如“为什么要慢晒”“哪一步会造成泥痕”“返修为什么要复洗退泥”。

### 4.3 NPC 见闻

| 字段 | 写法 |
| --- | --- |
| `id` | `intel-{npcShortName}-{topic}` |
| 归属 | NPC 的 `intel` 或百工志 `npc` 条目 |
| 触发 | 交谈、赠礼、授艺、活动、订单、家访 |
| 作用 | 解锁资料卡、影响结局回声、提示路线或工艺风险 |

NPC 文本要有“人味”和“地方信息”两层：一层表现性格，一层给玩家可用信息。

### 4.4 活动文本

| 字段 | 写法 |
| --- | --- |
| `id` | `{regionPrefix}-{place}-{activity}` |
| 阶段 | 开场、旺场、收场或等价三段 |
| 客群 | 至少 3 类，有不同需求 |
| 组合 | 至少 2 到 3 个货品组合 |
| 策略 | 至少 2 到 3 个经营策略 |
| 结算 | 钱、声望、路线稳定、订单、flags |

活动不是一次性答题。当前优先版的活动文本必须留下后续订单或地区反馈。

## 7. 入库验收

审核通过后，新增文本必须满足：

1. 每条文本能被搜索或条件解锁。
2. 有地区落点的文本必须只做路线提示，不触发直接迁移。
3. 工艺文本必须指向真实工序、缺陷或返修方向。
4. NPC 见闻必须至少能影响一个资料卡、flags、订单或结局回声。
5. 活动文本必须至少能影响一次收益、声望、订单或路线状态。
6. 所有新增 ID 必须通过引用守卫，不允许悬空。

## 8. 审核口径

策划审核时不需要检查代码是否已经补齐，但需要确认三件事：

1. 主轴五区的文本密度是否足够支撑美术开工。
2. 骨架六区的保留程度是否不会让世界观显得被砍掉。
3. 入库模板是否足够明确，能让后续程序阶段直接执行。
