## 1. 一句话项目定义

《百工归位》是一个以中国传统手工艺和地区生活为核心的 2D 俯视页游。玩家不是在菜单里排队玩小游戏，而是在一个由地区、材料、工坊、NPC、商路、天气、订单、作品和声望共同运转的世界里生活、学习、制作、返修、交付、陈列和成名。

最终体验目标：

- 看起来像一片真实的土地：11 个地区各有地貌、建筑、人物、道路、水路、资源点和生活密度。
- 玩起来像一门真实的手艺：每门核心工艺都从材料、工坊、师承、真实工序、失败返修和作品命运建立完整链路。
- 长线像一个页游：日程、天气、订单、商路、声望、NPC 关系、藏客回访、断供/护商和作品收藏要能支持持续游玩。

## 2. 当前状态对齐

### 2.1 已有可运行骨架

当前项目已经具备：

- React + TypeScript + Vite + Phaser 3 + Zustand 的前端架构。
- 纯函数引擎 `src/engine/reducer.ts`，规则结算集中在 engine。
- 11 个大地区、43 个小地区、50 门手艺、约 81 类资源/产物。
- 多槽本地存档、时段/天气/季节、田圃、订单、NPC 功能、地区声望、商路稳定、护商、断供、夜市、珍品阁、专属联作、结局报告。
- 地图编辑器试装入口 `http://127.0.0.1:5173/?editor=1`，可摆放地块/建筑/道具/单位并导出 `artisania-map-editor/v2` JSON。
- 自动化测试最近记录为 197 项通过，详见 [当前进度与规划差距盘点.md](当前进度与规划差距盘点.md) 与 [开发者文档.md](开发者文档.md)。

### 2.2 还没有完成的关键部分

当前不能误判为“只差贴图”：

- 大多数地区还缺真正手工布置的地图点位、出入口、工坊、资源点和动线。
- 大多数旗舰工艺还缺真实工序级交互，当前更多是数据工序与选项结算。
- 外地 NPC、节令活动、人物线、回访线和地区结局仍需要增密。
- 美术资源距离最终统一风格还很远，但代码组本轮把资源套用与最终点位美化降到最低优先级，只保留必要资源 key 和占位接入。
- 页游化后端、云存档、账号、异步社交、排行榜等还未启动。

### 2.3 当前文档封板口径

这次交接采用三个硬口径：

1. **全地区同步设计**：不再以单一地区作为前置样板。江南、巴蜀、岭南、黔滇、荆楚、赣鄱、徽州、京畿、三晋、雪域、西域都要按同一规格推进。
2. **整图优先于瓦片**：图像组先做完整地区母版和风格板，再拆 tiles、buildings、props、characters。
3. **真实工艺优先于小游戏**：代码组做的是“工艺交互系统”，不是套皮小游戏库。

## 4. 不可偏离的设计原则

### 4.1 对图像组

- 每个地区先做**整图母版**，不要一开始就拼瓦片。
- 地图要像真实生活空间：水边有船，码头有货，工坊有半成品，市场有摊，路口有告示，NPC 有站位理由。
- 角色造型服务职业和地方生活，不靠标签化服饰识别身份。
- 民族与宗教相关内容必须尊重资料语境，唐卡、苗银、艾德莱斯、藏纸、藏银等不能做成猎奇装饰。
- 地区之间要有统一世界观：纸白、墨线、朱砂点睛可贯穿，但主色、建筑剪影和地貌必须区分。

### 4.2 对代码组

- 游戏规则只进 `src/engine`，React 和 Phaser 不直接写数值规则。
- 新内容优先数据驱动，先补 `src/data` 与类型，再接 UI/场景。
- 工艺交互不是孤立小游戏，要与材料、工坊、NPC、失败返修、订单、作品实例和结局报告相连。
- 地图编辑器导出的 JSON 是人工布局输入，接入运行时时要保持数据结构稳定。
- 新增功能必须补测试，尤其是 reducer、存档迁移、数据守卫和跨区供应链。

### 4.3 对两组共同

- 任何 ID 一旦进入数据或素材路径，默认稳定，不随意重命名。
- 不用“先某一地区做完再说”的口径；可以按技术风险做验证，但内容规格按 11 区同步验收。
- 不把“资料真实”理解成百科堆砌。真实资料要转化成地图、人物、交互、作品和后果。


### 5.4 图像组验收标准

每个地区必须能回答：

1. 不看文字，能不能认出地区？
2. 工坊、资源点、市场、交通点是否在同一张土地里自然存在？
3. NPC 是否有职业道具、站位理由和地方气质？
4. 工艺道具是否能支持真实流程，而不是只当装饰？
5. 是否能拆成运行时资源，并进入地图编辑器复核？

### 6.3 首轮技术任务建议

1. **地图 JSON 接入运行时**
   首批样板已迁入 `src/data/mapLayouts/*.json`，通过 `runtimeLayoutFromEditorSnapshot` 从检入 JSON 资产生成 `RUNTIME_MAP_LAYOUTS`，再进入 `regionSpec.ts` / `StreetScene.buildRegion()`；11 个大地区目前各有一张代表小地区人工布局，支持人工道路、交互点、出入口、NPC 站位、玩家起点和简易碰撞装饰。下一步是把编辑器导出的完整 `artisania-map-editor/v2` JSON 文件、瓦片图层、季节变体和全部小地区精确点位接入同一管线。

2. **正式移动边界与通道守卫**
   保留大地图/区域图作为总览和调试入口；正式迁移必须通过街景通道、出口、商路节点或交互点触发。补充调试开关、提示清理和数据守卫，避免后续把“点击地图传送”误接成正式玩法。

3. **工艺交互规格数据化**
   参考 [全地区工艺交互模式规划.md](全地区工艺交互模式规划.md) 的 `CraftInteractionSpec`，先做类型草案和数据守卫。不要急着把所有交互做成 UI，先让数据能描述真实工序。
   - 进度 2026-06-12：已新增 `src/data/craftInteractions.ts`，11 个大地区各覆盖一门代表工艺，并继续扩展江南“龙泉青瓷”、临安“油纸伞”、太湖“缂丝”、巴蜀“蜀锦”、岭南“香云纱”、黔滇“苗银”、荆楚“楚漆”、赣鄱“景德镇瓷”、徽州“宣纸”、京畿“景泰蓝”和三晋“平遥推光漆器”规格；数据守卫会校验地区覆盖、工艺/小地区/工序 ID、师承 NPC ID 和返修方案引用。

4. **作品品质维度扩展**
   在 `ItemInstance` 或相关结构上承载材料纯度、形制、工序完整、火候/水候、纹样/线法、手感、故事痕迹等描述或标签。
   - 进度 2026-06-12：`ItemInstance` 已扩展 `qualityDimensions`、`defects`、`repairHistory`；`RUN_PROCESS` 会按工艺规格、跳过工序、材料品质和工艺四维写入维度与缺陷。
   - 进度 2026-06-13：`craftTechniques.ts` 与 `RUN_PROCESS.techniqueChoices` 已把制作前的工序手法接入规则层；`CraftPage` / `CraftExperienceModal` 会按真实工序展示“稳手细作 / 平衡推进 / 赶工压时”，并把选择写入工时、品相、焦点维度、作品鉴评、`craftTechniqueRecords` 和 `craft-technique:*` flags。当前仍是规则/UI 过渡层，后续美术与交互组可按同一 `CraftInteractionStage` 替换为锻打、淬火、运笔、牵经等实时动作小游戏。
   - 进度 2026-06-13：阶段诊断已进入作品实例。`CraftStageOutcome` 会为每个参与工序记录控制分、风险分、稳/平/险结果、焦点维度和来源步骤，背包“近作评鉴”会展示阶段诊断；后续视觉表现可按这些字段决定火候偏差、锤痕、线法松紧、釉面病根或师傅复核提示。

## 7. 两组协作方式

### 7.1 每个地区一个交付包

每个地区最终应形成同名包：

```text
regionId
  美术母版
  地图编辑器 JSON
  NPC/建筑/道具资源表
  工艺交互规格
  活动/订单/人物线数据
  测试清单
```

### 7.2 ID 对齐原则

所有交付物尽量使用当前代码中的地区 ID：

| 地区 | regionId |
|---|---|
| 江南 | `jiangnan` |
| 巴蜀 | `bashu` |
| 岭南 | `lingnan` |
| 黔滇 | `qiandian` |
| 荆楚 | `jingchu` |
| 赣鄱 | `ganpo` |
| 徽州 | `huizhou` |
| 京畿 | `jingji` |
| 三晋 | `sanjin` |
| 雪域 | `xueyu` |
| 西域 | `xiyu` |

图像资源、地图 JSON、文档表格和代码数据都优先复用这些 ID，避免后续接入时重命名。

### 7.3 评审顺序

每个地区建议按这个顺序评审：

1. 地区整图是否成立。
2. 子地区和动线是否清楚。
3. 工坊和工艺交互是否真实。
4. NPC 与建筑是否匹配。
5. 能否拆运行时素材。
6. 代码数据是否能承载这些内容。
7. 是否有测试和验收入口。

### M3 · 运行时接入样板

目标：验证管线，不改变全地区同步规格。

- 地图编辑器 JSON 进入运行时。
- 至少一个地区包跑通“整图拆分 -> 编辑器摆放 -> 运行时渲染 -> 交互触发”。

说明：M3 可以选择技术风险最低的地区做接入验证，但这不是内容优先级变化；11 个地区的规格仍同步推进。

### M4 · 全地区扩面

目标：把样板管线复制到 11 个地区。

- 每个地区都有可运行地图或可导入地图 JSON。
- 每个地区至少一门核心工艺交互进入可玩链路。
- 每个地区至少一组 NPC 事件、订单、联作或回访进入数据。
- 数据守卫覆盖全地区完整性。

## 9. 当前最大风险

| 风险 | 表现 | 应对 |
|---|---|---|
| 做成小游戏库 | 工艺只剩 QTE/打地鼠/按停 | 严格执行 [全地区工艺交互模式规划.md](全地区工艺交互模式规划.md) |
| 美术变成瓦片拼贴 | 地图没有整片土地感 | 先整图母版，再拆瓦片 |
| 11 区密度不均 | 有的地区像主城，有的像材料点 | 统一地区包和验收清单 |
| 代码规则散落 UI | Phaser/React 直接改数值 | 规则只进 engine，UI/场景只派发动作 |
| 资料堆砌不可玩 | 文档像百科，玩家无感 | 每个资料点都转成材料、工序、人物或后果 |
| 长线重复疲劳 | 玩家每次都被迫完整操作 | 首次/珍品深交互，日常可排程、委托、批量 |
